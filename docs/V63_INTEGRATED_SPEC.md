# Reverie v63 통합 시방서

최종 갱신: 2026-03-10

## 1. 문서 목적

이 문서는 Reverie Studio의 기능 요구사항, 품질 기준, 인수 기준을 정의한다.  
설계 변경은 허용하지만, 아래 시방 기준은 임의로 낮추지 않는다.

## 2. 제품 목표

- 팩 기반으로 채널/장르/스타일을 바꿀 수 있어야 한다.
- GUI에서 기획, 생성, 승인, 재시작을 안정적으로 수행할 수 있어야 한다.
- 생성 실패 시 처음부터 다시 하지 않고 마지막 완료 단계부터 이어갈 수 있어야 한다.
- 로컬 실행 환경에서도 상용 운영 수준의 로그와 검증성을 가져야 한다.

## 3. 범위

### 포함

- 시나리오 생성
- 썸네일 생성 및 승인
- TTS/자막 생성
- 이미지 생성
- Remotion 최종 렌더링
- 배치 큐
- 체크포인트 재개
- 팩 로딩/검증

### 제외

- 장르 하드코딩
- 평문 소스 무방비 배포
- 라이선스 없는 실행 허용

## 4. 기능 요구사항

### FR-01 팩 우선 구조

- 모든 프롬프트, 톤, 스타일 규칙은 팩 또는 팩 기반 설정에서 읽어야 한다.
- 코드 내부에 특정 장르 전용 분기를 새로 추가하지 않는다.

인수 기준:

- 새 장르 추가 시 코드 변경 없이 팩만으로 동작해야 한다.
- 프롬프트 키 누락 시 명시적 오류 또는 검증 경고가 나와야 한다.

### FR-02 GUI 생성 실행

- 사용자는 GUI에서 직접 생성, 대기열 실행, 미리보기 승인 후 생성을 수행할 수 있어야 한다.
- 각 경로는 동일한 파이프라인 옵션을 공유해야 한다.

인수 기준:

- 직접 생성, 큐 실행, 미리보기 승인 경로 모두 같은 공통 실행 helper를 사용한다.
- 진행률과 로그 콜백이 정상 전달된다.

### FR-03 체크포인트 재개

- GUI에 체크포인트 재개 옵션이 있어야 한다.
- 재개 옵션이 켜진 경우 마지막 성공 단계 이후부터 작업을 이어가야 한다.

인수 기준:

- 큐 작업에 재개 옵션이 저장된다.
- 직접 생성 경로에 재개 옵션이 전달된다.
- 미리보기 승인 후 생성 경로에 재개 옵션이 전달된다.
- 성공 시 체크포인트는 삭제된다.
- 실패 시 체크포인트는 유지된다.

### FR-04 취소와 복구

- 사용자가 생성 중단을 요청할 수 있어야 한다.
- 중단 또는 실패 후, 이전 산출물을 재활용할 수 있어야 한다.

인수 기준:

- 취소 후 새 작업이 이전 cancellation state를 물고 가지 않는다.
- 재시작 시 이전 단계 산출물을 다시 만들지 않는다.

### FR-05 배치 큐

- 작업별로 팩, 주제 모드, 썸네일 옵션, 업로드 공개범위, 재개 옵션을 저장해야 한다.
- 배치 큐 파일 저장은 원자적이어야 한다.

인수 기준:

- 큐 JSON 저장 중 프로세스가 끊겨도 파일 손상이 최소화된다.
- 작업별 옵션이 다음 실행에서 유지된다.

### FR-06 로그와 진단

- 장애 지점은 사용자가 로그만 보고도 최소한 “어느 단계에서 멈췄는지” 알 수 있어야 한다.

인수 기준:

- 최소 단계 태그: `thumbnail`, `tts`, `images`, `assemble`, `finalize`
- 재개 여부와 재개 단계가 로그에 남는다.

## 5. 비기능 요구사항

### NFR-01 안정성

- 예외를 삼키지 않는다.
- 사용자가 복구 가능한 오류와 개발자 대응이 필요한 오류를 로그에서 구분 가능해야 한다.

### NFR-02 성능

- 동일 입력 재생성 시 캐시 재사용 여지가 있어야 한다.
- 장시간 배치 시 temp 파일 누적을 제어해야 한다.
- 8GB VRAM 환경에서는 SD 입력 해상도, step, 렌더 동시성을 안전 한계 안에서 자동 제한해야 한다.

### NFR-03 보안

- API 키 하드코딩 금지
- 라이선스 우회가 쉬운 구조 금지
- 평문 팩/평문 배포 최소화

### NFR-04 테스트 가능성

- GUI를 띄우지 않아도 import/runtime/callback 경로를 검증할 수 있어야 한다.
- 핵심 옵션 전달은 mock 기반 테스트로 검증해야 한다.

## 6. 추가 경쟁력 시방

### CF-01 팩 검증기

- 필수 파일, schema, placeholder, asset 존재 여부를 자동 검사

인수 기준:

- 잘못된 팩을 배포 전에 차단
- 에러 메시지는 비개발자도 이해 가능

### CF-02 성과/비용 계측

- 프로젝트마다 단계별 소요 시간과 실패 원인을 기록

인수 기준:

- 최소 항목: plan, tts, image, render, total, fail_stage

### CF-03 운영자용 즉시 재시도

- GUI에서 마지막 실패 작업을 다시 불러와 재개할 수 있어야 한다.

인수 기준:

- 재시도 버튼 또는 최근 실패 작업 목록 제공
- 기존 JSON과 체크포인트를 재사용

### CF-04 하이브리드 실행

- 로컬 엔진 장애 시 대체 provider를 선택할 수 있어야 한다.

인수 기준:

- provider 정책은 코드 하드코딩이 아니라 설정 또는 팩 옵션으로 제어

## 7. 테스트 시방

자동 테스트 최소 기준:

- import/runtime 테스트
- 배치 큐 옵션 저장 테스트
- GUI 실행 helper의 옵션 전달 테스트
- 체크포인트 재개 플래그 전달 테스트

수동 테스트 최소 기준:

1. GUI에서 재개 옵션 ON 상태로 생성 시작
2. 중간에 외부 엔진을 끊어 실패 유도
3. 같은 프로젝트 재실행
4. 이미 끝난 단계가 재실행되지 않는지 확인
5. 성공 후 체크포인트 파일이 제거되는지 확인

## 8. 완료 정의

다음을 만족하면 v63 기준 구현 완료로 본다.

- GUI 3경로에서 체크포인트 재개 동작
- 주요 대형 파일 분해 1차 완료
- 병목 로그 수집 가능
- 문서 정본 3종 유지
- 비GUI 자동 테스트 통과
