## 개요

크몽 배포 후 6개월간 구현할 **6개 후속 기능**의 우선순위와 타임라인.

## 기능 1: 모션툰 실전 투입 (M1~M2)

### 왜 최우선인가?
- 정적 이미지 슬라이드쇼 → **캐릭터가 움직이는 영상**으로 체감 퀄리티 도약
- 코드 기반은 이미 90% 완성 (motiontoon_asset_pipeline, layered_cutout, RadioDrama.tsx)
- **경쟁사 대비 결정적 차별화** 포인트

### 구현 범위

**Tier 1: 즉시 연결 (M1 전반)**
| 모션 | 코드 위치 | 작업 |
|------|----------|------|
| Eye blink | motiontoon_asset_pipeline.py | Remotion 타이밍 연결 |
| Mouth flap | character_library_manager.py | 표정 스왑 자동화 |
| Idle breathing | layered_cutout.py | 미세 이동 파라미터 |
| Impact shake | RadioDrama.tsx MotionDirective | 활성화 + 트리거 |
| Snap zoom | RadioDrama.tsx | 활성화 |

**Tier 2: 추가 구현 (M1 후반~M2)**
| 모션 | 작업량 | 효과 |
|------|--------|------|
| 감정별 표정 자동 스왑 | 3일 | 슬픔→울음, 분노→노려봄 |
| 소품 애니메이션 | 5일 | 전화기, 문, 서류 |
| 컷 분할 리듬 | 3일 | 장면 전환 효과 |

**팩 설정 추가 (코드 수정 0줄 원칙 유지)**
```json
// settings.json에 motiontoon 블록 추가
"motiontoon": {
  "enabled": true,
  "eye_blink": true,
  "mouth_flap": true,
  "idle_breathing": true,
  "emotion_swap": true
}
```

### 완료 기준
- [ ] 3팩 모두 모션툰 ON 상태로 영상 3편 생성 → 시각적 개선 확인
- [ ] 크래시 0회
- [ ] 렌더링 시간 증가 50% 미만

### 예상 일정
- M1 1주~2주: Tier 1 모션 연결 + 테스트
- M1 3주~4주: Tier 2 시작
- M2 1주~2주: 3팩 팩 설정 + 풀 검증
- **총 6주**

## 기능 2: 자동 숏츠 추출 (M2~M3)

### 왜 높은 우선순위인가?
- YouTube 숏츠 = 채널 성장의 가장 빠른 경로
- 장편 10분 영상에서 **하이라이트 30~60초 자동 추출** → 숏츠용 9:16 렌더
- 영상 1편 생산 → 장편 1편 + 숏츠 2~3편 = **콘텐츠 3배 증식**

### 구현 방식
1. **하이라이트 자동 선택**: ScriptQualityGate의 tension score 활용
   - 각 턴의 감정 강도 분석 → 상위 3개 구간 자동 선택
2. **9:16 프레이밍**: RadioDrama.tsx에 이미 구조 존재
   - 캐릭터 중심 크롭 + 자막 재배치
3. **숏츠 전용 렌더**: Remotion composition 추가
4. **팩 설정**으로 숏츠 스타일 제어 (코드 수정 0줄)

### 완료 기준
- [ ] 장편 1편 → 숏츠 2편 자동 생성
- [ ] 9:16 비율, 30~60초 범위
- [ ] 자막/캐릭터 크롭 정상

### 예상 일정
- M2 3주: 하이라이트 선택 알고리즘
- M2 4주~M3 1주: 9:16 Remotion composition
- M3 2주~3주: 통합 테스트 + 팩 설정
- **총 5주**

## 기능 3: 시리즈 연재 모드 (M3~M4)

### 왜 필요한가?
- 현재: 매 에피소드 독립 → 캐릭터/배경 무관
- 목표: **시리즈물** (에피소드 간 캐릭터/설정/스토리 연속성)
- 시청자 리텐션 극대화 (다음 편 기대감)

### 구현 방식
1. **시리즈 컨텍스트 파일** (series_context.json)
   - 등장인물 프로필, 관계도, 이전 에피소드 요약
   - ScenarioPlanner에 이전 맥락 주입
2. **캐릭터 ID 고정**: character_library에서 시리즈 전용 캐릭터 이미지 재사용
3. **이전 에피소드 참조 프롬프트**: 팩 prompts/에 `series_continuation.txt` 추가
4. **GUI**: 시리즈 관리 패널 (에피소드 목록, 캐릭터 관리)

### 완료 기준
- [ ] 3편 연속 시리즈 → 캐릭터 동일성 유지
- [ ] 스토리 맥락 이어짐 확인
- [ ] 시리즈 컨텍스트 자동 갱신

### 예상 일정
- M3 3주: 시리즈 컨텍스트 설계 + ScenarioPlanner 연동
- M3 4주~M4 1주: 캐릭터 ID 고정
- M4 2주~3주: GUI + 통합 테스트
- **총 5주**

## 기능 4: 팩 마켓플레이스 (M4~M5)

### 왜 중간 우선순위인가?
- 수익 다각화: 팩 판매 = 추가 수익원
- 생태계 구축: 사용자가 직접 팩 제작/공유
- 코드 수정 0줄 원칙의 궁극적 완성

### 구현 방식
1. **팩 스토어 UI**: GUI 내 팩 브라우징/다운로드
2. **Firebase Storage**: .revpack 파일 호스팅
3. **팩 메타데이터 API**: Firestore `/pack_store/{packId}`
4. **결제 연동**: 토스페이먼츠 (유료 팩)
5. **팩 제작 위자드**: 비개발자도 팩 만들 수 있는 GUI

### 완료 기준
- [ ] GUI에서 팩 목록 조회/다운로드 동작
- [ ] 신규 팩 2종 스토어 등록 (comedy, mystery)
- [ ] 결제 → 라이선스 → 다운로드 플로우

### 예상 일정
- M4 3주: Firebase Storage + 메타데이터 API
- M4 4주~M5 1주: GUI 팩 스토어
- M5 2주~3주: 결제 연동 + 팩 제작 위자드
- **총 5주**

## 기능 5: 다국어 더빙 (M5~M6)

### 왜 중간 우선순위인가?
- 해외 시장 진출 = 사용자 10배 확장 가능성
- 한국어 대본 → 영어/일본어 자동 번역 + 해당 언어 TTS
- 글로벌 YouTube 채널 운영 가능

### 구현 방식
1. **번역 파이프라인**: Gemini API 기반 대본 번역
2. **다국어 TTS**:
   - 영어: Bark / XTTS v2 (오픈소스)
   - 일본어: Style-Bert-VITS2
3. **자막 로캘**: Remotion 자막 폰트/위치 자동 조정
4. **팩 다국어 설정**: `"languages": ["ko", "en", "ja"]`

### 완료 기준
- [ ] 한국어 영상 → 영어 더빙 버전 자동 생성
- [ ] TTS 퀄리티: 자연스러운 영어 발화
- [ ] 자막 일치율 95%+

### 예상 일정
- M5 3주: 번역 파이프라인 + Gemini 연동
- M5 4주~M6 1주: 영어 TTS 통합 (XTTS v2)
- M6 2주~3주: 일본어 + 자막 로캘 + 테스트
- **총 5주**

## 기능 6: 클라우드 렌더링 (M6+)

### 왜 낮은 우선순위인가?
- GPU 없는 사용자 대상 → 시장 확대
- 인프라 비용 발생 (GPU 클라우드)
- M1~M5 기능이 먼저 안정화되어야 함

### 구현 방식
1. **렌더 서버**: RunPod / Vast.ai GPU 인스턴스
2. **잡 큐**: Firebase Queue + Cloud Functions
3. **클라이언트**: GUI에서 "클라우드 렌더링" 옵션 선택
4. **과금**: 편당 $0.7~1.0 원가 → 월정액 또는 편당 과금

### 완료 기준
- [ ] 클라우드에서 영상 1편 렌더링 성공
- [ ] 로컬 결과물과 동일 퀄리티
- [ ] 결제 → 렌더 → 다운로드 플로우

### 예상 일정
- M6 전반: 아키텍처 설계 + 프로토타입
- M6 후반~M7: 베타 테스트
- **총 4~6주 (MVP)**
